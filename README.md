# Задание 1. Анализ и планирование

### 1. Описание функциональности монолитного приложения

**Управление отоплением:**

- Пользователи могут удалённо включать/выключать отопление в своих домах.
- Система поддерживает только управление отоплением в доме путем регистрации датчиков при их включении и управлением этими датчиками.
- Самостоятельно подключить свой датчик к системе пользователь не может. Для этого нужен выезд специалиста по подключению системы отопления в доме к текущей версии системы.

**Мониторинг температуры:**

- Пользователи могут просматривать текущую температуру в своих домах через веб-интерфейс.
- Система поддерживает получение данных о температуре через запрос от сервера к датчику.


### 2. Анализ архитектуры монолитного приложения

Архитектура приложения представляет из себя монолит на Go с СУБД Postgres. Всё синхронно. Никаких асинхронных вызовов, микросервисов и реактивного взаимодействия в системе нет. Всё управление идёт от сервера к датчику. Данные о температуре также получаются через запрос от сервера к датчику.
- Язык программирования: Go
- База данных: PostgreSQL
- Архитектура: Монолитная, все компоненты системы (обработка запросов, бизнес-логика, работа с данными) находятся в рамках одного приложения.
- Взаимодействие: Синхронное, запросы обрабатываются последовательно.
- Масштабируемость: Ограничена, так как монолит сложно масштабировать по частям.
- Развертывание: Требует остановки всего приложения.

### 3. Определение доменов и границы контекстов

Домены, которые предлагается выделить.

- домен "Управление Устройствами"
  - поддомен "управления пользователями" -
    создание учетных записей пользователей
  - поддомен "управления устройствами" -
    сохранение базовой информации об устройствах, как только она появилась или изменилась остальные домены извещаются об этом через сообщение 'sensor info changed'
  - поддомен "комманд" -
    возможность отправить on-line комманду устройсву - 'установи температуру', 'открой ворота', об изменнеии состояния устройств в результате этой команды узнаем через домен Мониторинга
    - контекст Устройства -
    здесь устройство рассматривается с полным набором основной информации о нем
- домен "Выполнения сценариев устройств" 
  - поддомен "управления сценариями устройств" -
    создание-редактирование сценариев через REST API
  - поддомен "процессинга сценариев устройств" -
    core-часть срабатывающих сценариев, получает сообщения об изменениях базовой информации самих устройств (создано новое, удалено, и т.д.), и получает сообщения 'sensor state changed' об изменении состояния устройства от домена Телеметрии (температура изменилась)
    - контекст Сценарии -
      помимо сохранения самого сценария, здесь информация об устройстве также храниться, но только часть, необходимая для выполнения сценария
- домен "Мониторинга устройств"
  - поддомен "получения телеметрии" -
    периодический опрос всех устройств об их актуальном состоянии. список устройств поддерживается при получении сообщения 'sensor info changed'. отправляет сообщение 'sensor state changed' при обраужении нужных изменений в состояниях устройств.
  - поддомен "просмотра телеметрии" -
    API, которое отдает актуальное состояниее устройства или состояния за период времени. поддержка фильтрации и поиска.
    - контекст Телеметрия
      помимо телеметрии поддерживаем актуальным спсиок устройств. для каждого храним только нужную информацию о нем. возможно здесь используется специализированная БД.


### **4. Проблемы монолитного решения**

- Высокий риск ошибок. 
  Программирование настройки сценариев одного типа устройства может повлиять на сценарии другого типа из-за общей кодовой базы. Это увеличивает риск ошибок и требует дополнительных ресурсов на тестирование и отладку всей системы.
  - Длительные циклы разработки и развёртывания. 
    Нужно добавить много нового функционала. Самостоятельную регистрацию устройст конечным пользователем, настройки сценариев для различных типов устройств. 
    Есть риск, что компания не успеет выполнить условия тендера к назначенным срокам. Компании нужно ускорить разработку новых функций.
- Трудно управлять командой. 
  Сейчас все компоненты приложения объединены в единый блок. Чтобы внести изменение или добавить функциональность, нужно внести изменения во всё приложение. Задачи, которыми занимаются разные команды, могут блокировать друг друга.
- Трудно масштабировать отдельные компоненты системы. 
  Объёмы данных о устройствах возрастут, появяться новые типы устройств, но масштабировать систему уже на этом этапе практически невозможно.

# Задание 4. Создание и документирование API

### 1. Тип API

Тип API используемый для взаимодействия микросервисов. Объяснение решения.

В первую очередь необходимо выделить интерфейс пользователя в отдельное веб-приложение,
далее, настроить его, чтобы оно обращалось к API gateway, а не напрямую к API, 
Это дает возможность маршрутизации запросов к разным микросервисам.
Внешний протокол взаимодействия между веб-клиентом и API gateway будет REST как стандарт в индустрии. Перечислим его плюсы:
 - Независимость. Базируется на HTTP и JSON, которые понимают разные платформы, что позволяет выбрать для реализации отдельного микросервиса свой язык и стек.
 - Единообразие. Выражается в использовании стандартных методов HTTP (GET, POST, PUT, DELETE) для операций.
 - Масштабируемость. Методы реализуются по принципу без хранения состояния. Таким образом любой из экземпляров сервиса может обработать любой запрос.
 - Кэширование. Получаем из коробки на стандартных решениях.

Внутренний протокол синхронных вызовов между микросервисами будет также REST, но на данном этапе внутренние вызовы между сервисами не используются.

Для асинхронного взаимодействия будем использовать брокер сообщений.
Он позволяет слабо связывать сервисы путем извещения получателя о событии. 
Позволяет сохранять сообщения в БД брокера, что обеспечивает частичную работоспособность системы - получатели могут не обрабатывать сообщения некоторое время.
Позволяет буфферезировать сообщения для отложенного и растянутого во времени потребления.

# Задание 5. Работа с docker и docker-compose

Перейдите в apps.

Там находится приложение-монолит для работы с датчиками температуры. В README.md описано как запустить решение.

Вам нужно:

1) сделать простое приложение temperature-api на любом удобном для вас языке программирования, которое при запросе /temperature?location= будет отдавать рандомное значение температуры.

Locations - название комнаты, sensorId - идентификатор названия комнаты

```
	// If no location is provided, use a default based on sensor ID
	if location == "" {
		switch sensorID {
		case "1":
			location = "Living Room"
		case "2":
			location = "Bedroom"
		case "3":
			location = "Kitchen"
		default:
			location = "Unknown"
		}
	}

	// If no sensor ID is provided, generate one based on location
	if sensorID == "" {
		switch location {
		case "Living Room":
			sensorID = "1"
		case "Bedroom":
			sensorID = "2"
		case "Kitchen":
			sensorID = "3"
		default:
			sensorID = "0"
		}
	}
```

2) Приложение следует упаковать в Docker и добавить в docker-compose. Порт по умолчанию должен быть 8081

3) Кроме того для smart_home приложения требуется база данных - добавьте в docker-compose файл настройки для запуска postgres с указанием скрипта инициализации ./smart_home/init.sql

Для проверки можно использовать Postman коллекцию smarthome-api.postman_collection.json и вызвать:

- Create Sensor
- Get All Sensors

Должно при каждом вызове отображаться разное значение температуры

Ревьюер будет проверять точно так же.


# **Задание 6. Разработка MVP**

Необходимо создать новые микросервисы и обеспечить их интеграции с существующим монолитом для плавного перехода к микросервисной архитектуре. 

### **Что нужно сделать**

1. Создайте новые микросервисы для управления телеметрией и устройствами (с простейшей логикой), которые будут интегрированы с существующим монолитным приложением. Каждый микросервис на своем ООП языке.
2. Обеспечьте взаимодействие между микросервисами и монолитом (при желании с помощью брокера сообщений), чтобы постепенно перенести функциональность из монолита в микросервисы. 

В результате у вас должны быть созданы Dockerfiles и docker-compose для запуска микросервисов. 

### **Что сделано**

1. В папке MVP создан новый docker-compose
он поднимает вторую базу для нового сервиса telemetry-api
2. поднят брокер сообщений kafka для получения информации от монолита,
   его адрес вынесен в env QUEUE_ADDRESS.
   образ монолита используется тот же самый, но 
   в монолит добавлен env var флаг MVP, если он 1, то работает функционал интеграции с новым сервисом:
   во время создания или удаления сенсора отправляется сообщение с информацией о сенсоре в kafka.
   telemetry-api получает сообщение и добавляет/удаляет сенсор у себя,
   он хранит не всю информацию о сенсоре, а только необходимую ему.
3. монолит без установленной env MVP продолжает работать по старому и доступен в интеграции с сервисом temperature-api из предыдущего задания.
